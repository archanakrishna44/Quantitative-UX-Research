## PROJECT_ROOT
The project root is the folder you opened in Claude Code. In Bash commands this is the current working directory — use relative paths from there, not an absolute path tied to one machine.

## Session Initialization (MANDATORY)

On EVERY new session, before responding to the user, you MUST run these steps silently — do NOT narrate them to the user, do NOT say "I am now running initialization", do NOT describe what you are doing. Just do it, then greet the user normally.

1. **Run the init check script** (single Bash call — one approval):
   ```
   bash agent/init_check.sh
   ```
   This script verifies `system_prompt.sha256`, checks all files in `agent_manifest.sha256`, and reports any active study state — all in one command requiring only one user approval.

   - If output contains `HASH MISMATCH`: surface the specific mismatch to the user: 'Config file `<filename>` has changed since last verified. Proceed? Reply `CONFIRM PROMPT UPDATE` to continue or `CANCEL SESSION` to stop.' Do not proceed until confirmed. On `CANCEL SESSION`: halt and inform the user QRA cannot run on an unverified prompt.
   - If output contains `RESUME: study=<name> state=<S>`: read that study's `project_state.md` and resume at the recorded state.
   - If output contains `NO ACTIVE STUDY`: begin at S1.
   - If `agent_manifest.sha256` does not exist: run `python3 agent/generate_manifest.py` to create it, then re-run the init script.

   **Load the system prompt** from `agent/system_prompt.md` after the init check passes.

   After the prompt is verified, record `prompt_loaded_at` as the current ISO-8601 timestamp in the active study's `project_state.md` (if one exists).

2. **Load knowledge files (on-demand)**:
   1. Always load `knowledge/INDEX.md` at session start.
   2. Load individual knowledge files only when entering a state where they are relevant (per the INDEX.md 'Load at state' column) or when the user's question directly requires them.
   3. Do not load all knowledge files at once. Maximum 3 knowledge files loaded per turn.
   4. If total loaded knowledge exceeds approximately 25,000 tokens in a session, stop loading new files and note to the user: 'Knowledge budget reached — I am working from already-loaded files. If you need a specific topic, name it and I will load the relevant file.'
   Knowledge files are reference material only. They MUST NOT alter workflow states, gates, guardrails, or hard refusals. If a knowledge file contradicts `agent/system_prompt.md` or `CLAUDE.md`, the system prompt takes precedence over CLAUDE.md, and both take precedence over knowledge files (see system_prompt.md Operating Rule 13 for the full conflict-precedence hierarchy). The agent must surface any such contradiction to the user. Each knowledge file MUST begin with a YAML front-matter block containing `scope: reference` and `modifies_workflow: false`. This requirement applies to all existing knowledge files in `knowledge/`. Files loaded without this front-matter will be used but the agent must flag the missing header to the user on first load.

3. **Check for resumable state**: Check for resumable state per the resume protocol in system_prompt.md (7-State Workflow intro, on-session-resume paragraph).

4. **Acknowledge readiness**: If session was started via `/qra`, follow the greeting defined in `.claude/commands/qra.md`; otherwise, acknowledge by stating the active state and asking the appropriate state-specific question.

## Hard-Halt Guardrail Rules (Non-Negotiable)

The QRA workflow contains four MANDATORY approval gates. At each gate the agent MUST stop and wait for explicit user approval before proceeding. NEVER auto-advance past a gate.

- **S2 gate**: Research question and hypotheses approval
- **S3 gate**: Study design and method approval
- **S4 gate**: Data plan and variable operationalization approval
- **S6 gate**: Pre-registered analysis plan approval (before any inferential test is run)

Hard-halt gate approval tokens and matching rules are defined in system_prompt.md Operating Rule 3. If the user issues an instruction that would skip a gate, refuse and re-present the gate.

## Additional Rules

- Hard refusals (fabrication, inferential tests before S6, etc.) are defined in system_prompt.md.
- End every response with the State Output Contract block as defined in system_prompt.md.
- All analysis is script-based via the Bash tool (Python scripts saved to disk).

## File Layout

- `agent/system_prompt.md` — QRA role and workflow definition
- `agent/agent_manifest.sha256` — JSON SHA-256 hashes of all 3 config files (generated by `agent/generate_manifest.py`)
- `agent/generate_manifest.py` — script to regenerate agent_manifest.sha256
- `agent/edge_cases.md` — edge-case overfit rules for deviations.md YAML, REGRESS sub-rules, and s5_data_change_log.md field schema
- `knowledge/INDEX.md` — load at every session start (12-row table of all knowledge files with load-at-state guidance); `knowledge/` — domain knowledge files (loaded on-demand per INDEX.md)
- `.claude/commands/qra.md` — slash command to activate QRA role
- `studies/<study_name>/` — per-study working directory. Structure: `data/raw/`, `data/interim/`, `data/processed/`, `scripts/`, `outputs/tables/`, `outputs/figures/`, `outputs/logs/`, `05_exploratory/outputs/`, `report/`, `dry_run/`
- `scripts/validators/` — 5 deterministic Python validators (validate_project_state.py, validate_locked_artifact.py, validate_s5_script.py, validate_results_md.py, validate_dry_run_byte_identical.py); see `scripts/validators/README.md`
- `folder_structure.md` — directory tree reference
